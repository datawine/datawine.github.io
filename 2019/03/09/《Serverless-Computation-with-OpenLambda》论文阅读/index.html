introduction

数据中心和其上软件平台的发展经历了几个阶段

  • 硬件
  • 基于硬件的虚拟化(hardware-based virtualization)
  • 基于容器的虚拟化(containers, combined with distribution tools like dockers)

但是现有的服务器架构让服务器难以configure和manage，服务器的启动时间也限制了可扩展性。

Instead of thinking of applications as collections of servers, developers instead define applications with a set of functions with access to a common data store.

一个例子就是Amazon’s Lambda。Lambda模型有几个优点

  • 开发者不需要关心服务器管理，因为云服务管理者管理公用的服务器池中来自不同用户的lambda handler。
  • handler通过跨函数共享运行环境，让代码中关于特定应用的部分更少，更容易在集群的任何一个地方运行。
  • 扩展快速。

lambda background

选取了AWS lambda云平台。

programming model

lambda模型采用回调函数的形式。于是这里讨论RPC。

开发者选择运行环境，上传代码，指定管理事件的函数名称。开发者就获得了一个URL，通过这个URL可以在客户端发送RPC请求。

lambda函数是无状态的。

优点

  • 扩展速度快

  • 易于扩展配置

lambda workloads

分析google gmail这种基于rpc的服务，这里我们可以分析lambda的负载。

通过分析可以看到有两种rpc请求：非常长的和非常短的。长rpc经常被用来作为long-pooling技术的一部分，服务器想要返回给客户端数据的时候，在这个长rpc中返回即可。

一些结论：

  • 应用程序应该多执行长rpc请求。
  • 实现细粒度的accounting
  • 长rpc会阻塞进程，导致空闲handler占中大部分时间

research agenda

下面展示几个lambda模型的问题。

execution engine

lambda的核心是一个执行handler的沙盒。AWS lambda使用容器，但是和server-based的架构不同，AWS lambda在不同的instance之间共享服务器和运行环境。

在低负载的时候，虽然AWS lambda尽可能用一个容器执行多个handler，但是latency仍然是Elastic BS的十倍。于是接下来讨论一些trade off。

首先，需要说明一个处理请求的容器必须是running状态，否则就会是paused或者stopped状态。

根据实验，一个容器从paused到running状态只需要1ms，但是重新建立一个容器或者从 stopped到running状态则需要100ms。但是让容器保持paused状态又需要大量的内存（和running需要的内存差不多）。

所以这里是一个需要研究的地方。

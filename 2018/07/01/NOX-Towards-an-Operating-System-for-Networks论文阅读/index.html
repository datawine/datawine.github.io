NOX: Towards an Operating System for Networks论文阅读

Jul 1, 2018 | 阅读

This paper introduce NOX, a network operation system.

A network operation system does not manage the network itself; it merely provides a programmatic interface.

Such a network operating system represents two major conceptual departures from the status quo. First, the net- work operating system presents programs with a centralized programming model; programs are written as if the entire network were present on a single machine. Second, programs are written in terms of high-level abstractions (e.g., user and host names), not low-level con- figuration parameters (e.g., IP and MAC addresses).

NOX overview

Components

  • a set of switches(supporting the OpenFlow switch abstraction) and one or more network-attached servers
  • The NOX software, along with the management applications

Granularity

For observation, NOX’s network view includes the switch-level topology; the lacation of users, hosts, middleboxes, and other network elements; and the services being offered. The view includes all bindings between names and addresses, but does not include the current state of network traffic.

For control granularity, NOX choose flows control. That is, once control is exerted on some packet, subsequent packets with the same header are treated in the same way.

Switch Abstraction

NOX uses the OpenFlow switch abstraction with entries of the form:

For each packet matching the specified header, the counters are updated and the appropriate actions taken. If a packet matches multiple flow entries, the entry with the highest priority is chosen. The basic set of OpenFlow actions are: forward as default (i.e., forward as if NOX were not present), forward out specified interface, deny, forward to a controller process, and modify various packet header fields (e.g., VLAN tags, source and destination IP address and port).

Operation

NOX applications use these flow-initiations and other forwarded traffic to (i) construct the network view (observation) and (ii) determine whether to forward traffic, and, if so, along which route (control).

Scaling

Our confidence in the scalability of NOX follows from considering the spectrum of timescales and consistency requirements. In terms of timescales, NOX processing occurs at three very different rates:

• Packet arrivals: this is on the order of millions of arrivals per second for a 10Gbps link.

• Flow initiations: with NOX’s definition of flow (which is typically more persistent than NetFlow’s definition), the flow-initiation rate is typically one or more orders of magnitude less than the packet arrival rate.

• Changes in the network view: in our deployment expe- rience this is on the order of tens of events per second for networks of thousands of hosts.

In terms of consistency, the only network state that is global (i.e., must be used consistently across the controller processes) is the network view; this consistency requirement arises because applications use data from the network view to make control decisions, and those decisions should be the same no matter to which controller instance the flow has been sent. In contrast, since neither packet state nor flow state are part of the network view, they can be kept in local storage (i.e., packet state in switches, and flow state in controller instances).

Implementation Status

The network view is a set of indexed hashtables.

Programmatic Interface

Events

To cope with change events, NOX applications use a set of handlers that are registered to execute when a particular event happens.

Events handlers are executed in order of their priority. A handlers return value indicates to NOX whether to stop execution of this event, or to continue by passing the event to the next registered handler.

Network View and Namespace

To perform conversions from high-level name into low-level address, high-level declarations can be “compiled” against the network view to produce low-level lookup functions that are enforced per-packet. These functions are recompiled on each change to the network view.

Because the network view must be consistent and made available to all NOX controller instances, NOX applications should only write to it when a change is detected in the network, and not for every received packet.

Control

Management applications exert network control through OpenFlow.

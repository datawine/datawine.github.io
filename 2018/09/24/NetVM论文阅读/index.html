NetVM

论文的名字是《NetVM: High Performance and Flexible Networking using Virtualization on Commodity Platforms》

Inroduction

一方面，Middlebox这种集合多种功能的软硬件包常常需要有high cost，所以作者希望可以通过可以灵活部署的virtual machines来进行优化，另一方面，SDN的模式在how data are forwarded方面更加灵活，但是它在control plane层面上的控制让它在data plane上仍然使用特有的硬件，让能够virtualization的NF变少了。此外，DPDK通过在绕过内核减少了包的拷贝，但是它并不能单独支持非常完善的virtualization。综上，开发了NetVM。其具有

  • 一个virtualization-based platform
  • 一个shared-memory framework，基于dpdk
  • 一个hypervisor-based switch能够通过state或者data动态调整流的目的地
  • 一个支持高速inter-VM通信的结构
  • 一个security domain能够限制限制packet在信任的VM间通信

Background and Motivation

Highspeed COTS Networking

影响performance的主要有两点：一、因为进程调度的原因，如果放在内核态，来了新的packets会引发中断等等内核事务，处理起来非常耗时；二、现在的os都是将内核态的packet拷贝到用户态，导致了大量的耗时。（以上通过dpdk来bypass解决，通过大页共享的方式）

但是dpdk不提供完整的框架来组建和interconnect复杂的NF。

此外，dpdk的SR-IOV（single root io virtualization）通过建立有一个L2 Switch来完成交换，这意味着一方面同一个switch下的VM传递需要新的拷贝，另一方面不同Switch的VM可能还需要go out of the host and come back，这就提出了NetVM的新模式。

Flexible Network Services

现在的dpdk平台提供的是L2层面的switch，很多情况下比如COTS platform被当成middlebox或者是路由，MAC地址就一样了，所以引入了hypervisor的模块进行更加详尽的switching。

Virtual Machine Based Networking

vm带来很多便捷的特点。

System Design

上图给了一些比较。

左边是传统的结构，packets到达NIC然后被复制到hypervisor，然后一个虚拟的switch通过L2或者更高层的switch，决定具体发给哪个VM，然后通知相应的虚拟NIC。包含包的memory page被拷贝或者granted给Guest OS，然后OS在拷贝到user application。

中间那个是dpdk采用的，SR-IOV，先在NIC做L2层的switching，然后data直接被拷贝到user space。但是因为到哪个VM是由NIC静态配置好了的，而且除了L2层的MAC之外一般不用比的做switch的根据，所以可扩展性不好。

右边是提出的相关理念。一方面用user space的hypervisor来做switch，一方面用shared memory减少拷贝到Guest的操作。

Zero-Copy Packet Delivery

上图给了如何实现zero-copy。主要有两个部分：一是小的内存共享区域，由每个VM和hypervisor共享，用来传递packet descriptor，二是大的内存共享区域，由trusted的VM群共享，允许直接读写packet。

NetVM核心，作为一个dpdk enabled的user application，从NIC读取包，进行分析，然后插入descriptor。不同的VM通过一个role number区分。descriptor包括mbuf location，huge page offset以及动作（transmitting、forwarding）和相关的下一步VM的role number。descriptor是需要被拷贝的。

Lockless Design

锁的开销是很大的，为了减少锁的使用，采用了多个队列和多个核去处理，并且NIC要求具有RSS的队列之间负载均衡的能力。

每次运行的时候，hypervisor的一个核独立的从NIC中receive一个packet，在队列尾加一个descriptor，对应的guest os从队列头读一个descriptor，处理packet，处理完在transmit的队列尾加一个descriptor，而hypervisor从transmit的队列头读取descriptor完成transmit。

注：每个核都有自己的queue。

NUMA-aware Design

因为涉及到多核，和numa就有些类似了，如果cache撞了，就会时间爆炸。NetVM为了避免这一点，内存空间被socket平均分配，NetVM在hypervisor同时创建一样多的thread，每个处理一个socket，guest os中的thread也类似。这保证了一个packet在处理的时候，永远呆在一个socket的分配的memory bank里，cacke lines will never need to be passed between sockets。

Huge Page Virtual Address Mapping

有以下问题

  • 内存布局只有hypervisor知道，那么guest os怎么找到共享内存区域的packet位置呢？
  • 找的过程越快越好

NetVM的第一种解决方法是：通过模拟的PCI，将内存块map成一个连续的区域。如下图

就算在guest os中的huge page是连续的，但是hypervisor的虚拟地址空间的地址对于guest os没有意义，遍历一边太naive了，NetVM用了only bit operation和precompute lookup tables。

计算的公式是这样的OFFSET(P)=HIGH(HMAP[IDMAP(P)])|LOW(P)

其中

  • LOW(P)为P地址的低30位
  • IDMAP(P)为P地址的高8位
  • HMAP[x]为P地址高8位为x时的huge page编号
  • HIGH(y)为huge page编号为y的对应于guest os的内存起始位置。

Trusted and Untrusted VMs

VM被创建的时候就被分组。dpdk可以有一个简单的分类，决定一个packet被拷贝到哪一个memory poll里。

Implementation Details

NetVM Manager

NetVM Manager和QEMU之间有一个socket-type chardev的连接，来通知双方每一个VM的建立。同时NetVM Manager还保存trust group和switching rules。

NetVM Core Engine

NetVM Core Engine是一个dpdk userspace application running in the hypervisor。它有两个功能：接收packet，然后传给VM；与NetVM Manager通信，同步新VM信息。

Emulated PCI

QEMU和KVM不支持直接在hypervisor和VM之间共享内存，于是通过Emulated PCI重定向内存地址。

每个VM有自己的private shared memory作为ring buffer来和hypervisor交换descriptor，还有一个大的一起共享的huge page的空间，这段空间在hypervisor中可能不连续，但是在VM中是连续的。

NetLib and User Applications

NetLib实际上一个adapter，user applications提供一些基本配置，包括核的个数和回调函数。收到包的时候回调函数就会被调用，然后决定下一步干啥。如下图

OpenNetVM

论文的名字是《OpenNetVM: A Platform for High Performance Network Service Chains》

Introduction

因为low level tool让我们能够建造network function，所以我们需要一个平台能够提供higher level的abstraction。特质包括

  • 一个轻量和高效的NF管理框架
  • 框架和NF都应该能够控制packet的传输
  • NF必须是独立、容易部署、容易开发的

OpenNetVM提供了相关的

  • 基于容器的NF
  • NF和flow management
  • Efficient IO
  • 可扩展性和优化

OpenNetVM architecture

主要结构如下图

NF Manager

NF Manager通过管理NFs和routing packets来维持网络状态。一共由以下三个部分组成：

Memory Management

OpenNetVM启动的时候，Manager线程用dpdk库来分配huge pages。这些内存被CPU sockets平均分配，保证了相关packets只在同一个socket下运行。

为了让Manager和VM都可以用同一套虚拟内存，NF在启动的时候从配置文件中找到相关配置，然后做好映射。就不用emulated PCI了。

NF Management

NF Manager通过一个message ring buffer来获得NF的消息。NF启动或者关闭的时候通过ring buffer来通知Manager。

Packet Flow

通过RSS（Receive Side Scaling）hash，packet到达某一个NIC端口。packet的内容直接放到shared memory里面，packet descriptor被拷贝到一个和socket和core绑定的RX线程的队列上。RX线程将他们发送到NF上。TX也同理。

Network Functions

NF通过packet descriptor和回调函数来处理包。

Flow Management

Service Type

NF开始的时候，会向manager声明自己的服务类型。Service ID表示服务类型，可能有多个NF有相同的Service ID，这时需要通过Instance ID来区分。

Service Chains

实际上就是default rules。

Flow Director and Flow Tables

NF Manager中包括一个flow table，将flow map到service chain上。

OpenNetVM includes an SDN NF which will contact an SDN controller via the OpenFlow protocol to determine the desired service chain for each flow. The SDN NF then uses the Flow Director API to configure the manager’s flow table. This approach keeps the manager as simple as possible; it
merely enforces the flow table rules, while allowing other applications to provide the policy to set them.
